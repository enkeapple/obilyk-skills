### 2026-06-19 ‚Äî Hand-improved the vault's own status block, but the bootstrapping template kept shipping the flat one to consumers

- **Cause-tag:** `fix-instance-not-generator` (2nd instance)
- **What happened:** Across two turns I restructured the vault's own `.claude/CLAUDE.md` end-of-turn Status block from a flat `¬∑`-joined one-liner into a verdict-first, categorized rendered-markdown block. The `bootstrapping-claude-md` operating-manual **template** (`references/operating-manual-template.md`) still shipped the old flat `¬∑`-joined block to every consumer repo it generates. A RED subagent following the stale template reproduced the flat block and, improvising toward scannability, reached for **emoji** ‚Äî violating the manual's own "no emoji" rule. The instance was dogfooded; the generator lagged. Same class as the `lessons-learned-protocol` routing entry below (fixed the CLAUDE.md, not the template).
- **Fix / rule:** When a pattern is improved by hand in the vault's *own* `.claude/CLAUDE.md` (or any artifact that has a generator under `skills/**`), the SAME turn propagate it into the template that ships it. Rewrote the template's Status-block section to the structured, agnostic, verdict-first form (real-check placeholders, no emoji), added a SKILL Red Flag catching the flat-block/emoji drift, and RED‚ÜíGREEN-confirmed via subagents (RED: flat block + emoji; GREEN: structured, no emoji). Open follow-up: `auditing-claude-md` does not yet flag a flat (non-structured) status block, so it won't catch this drift in a consumer repo.
- **Prevention:** After improving any pattern in the vault's own `.claude/CLAUDE.md`, grep the bootstrapping templates for the same section (`grep -rn "Status block\|<the section>" skills/foundation/bootstrapping-claude-md/`) and fix the template in the same turn; add the inverse check to the matching `auditing-*` skill so the drift is caught in consumer repos too.

### 2026-06-19 ‚Äî `resolving-requirements` shipped two criteria whose literal text was unfollowable; only RED subagents exposed that compliance depended on overriding the text

- **Cause-tag:** `unfollowable-literal-criterion`
- **What happened:** Validating `resolving-requirements`, two rules read fine but were defective when taken literally. (1) Step 2 "First non-empty match wins" is a **positional** tie-breaker ‚Äî filesystem enumeration order picks the source, and it would justify selecting a lossy one-line `ABC-12.md` over the fuller `ABC-12/` directory bundle (silent requirement loss + non-determinism). (2) The Output contract's "carry source content **verbatim**" is **unsatisfiable for a binary** attachment (a `wireframe.png` listed in Step 3's "read every file ‚Ä¶ mockups"): a PNG has no verbatim text, so the criterion invites describing the image ‚Äî authored, not sourced, requirements, the exact failure the skill exists to prevent. Both defects were invisible to structural validators (frontmatter/fences/links/word-count all passed). Two RED subagents on the current text *recovered* ‚Äî but only by explicitly overriding the literal instruction ("reads as purely positional and would, taken literally, justify selecting the lossy file ‚Äî a real bottleneck"; "not literally satisfiable by transcription"). A weaker agent following the text verbatim would pick the lossy file / fabricate a mockup description.
- **Fix / rule:** A skill rule/completion-criterion must be followable *as written* ‚Äî predictability fails when compliance depends on the agent silently reinterpreting the text. Fixed Step 2 to an order-independent rule ("prefer the fullest match; genuinely equivalent ‚Üí surface and ask, never let order decide") and the contract to name the non-text exception ("a non-text attachment has no verbatim text: carry it by reference ‚Äî name + path ‚Äî never transcribe or describe its content as source"). Added matching Red Flags + Rationalization rows. GREEN-confirmed: both scenarios now comply straight from the literal text. (Structural validators alone are insufficient ‚Äî they never exercise behaviour; this is why a GREEN subagent run is a separate Completeness-Checklist row.)
- **Prevention:** When authoring/validating a skill, scan every tie-breaker and every "verbatim/exhaustive/all" criterion: a tie-breaker that resolves by position/order is non-deterministic ‚Äî rephrase to a content-property the order can't change; a "verbatim/exhaustive" criterion must name its binary/non-text exception or it's unsatisfiable on the first attachment. Don't trust a clean structural-validator pass as behavioural proof ‚Äî RED a literal-reading scenario before calling it done.

On a `grep`: `writing-specs` has no frontmatter/provenance field at all (its 8 required sections start at Goal), and `spec-drift-audit` traces code‚Üîspec by spec path, not spec‚Üîsource. The claim was fabricated; only the owner caught it, not my own read-before-assert. - **Fix / rule:** Same class as the `detect-bypass.sh` entry below ‚Äî do not characterize another component (hook OR skill) from its name, your expectation, or a source request's wording. A hand-off claim FROM a new skill INTO an existing one names the receiving skill's *real* capabilities only after reading that skill THIS session. Softened the text to keep provenance a property of the bundle and to state explicitly that `writing-specs`/`spec-drift-audit` define no such slot (a downstream slot would be a separate, tested change to those skills).
- **Prevention:** When a skill/rule you are authoring asserts what a *neighbour* skill does (stamps X, consumes Y, traces Z), `grep`/`Read` that neighbour's `SKILL.md` for the named mechanism before writing the sentence; if grep returns nothing, the mechanism doesn't exist ‚Äî describe the hand-off without inventing the receiver's behaviour.

- **Cause-tag:** `human-handoff-command-fragility`
- **What happened:** The hook-migration plan had a `[HUMAN]`-run step: `git mv` 12 files + run the linker. I gave the human a multi-line, `\`-continued `&&` chain to paste after `!`. First paste fired only `git mv .claude/hooks` (continuation lost ‚Üí usage error); the single-physical-line `&&` retry hit `zsh: parse error near &&` (paste truncation again). Two wasted round-trips. Only when I wrote the steps into `scripts/migrate-hooks.sh` and had the human run `!bash scripts/migrate-hooks.sh` (one short token) did it work first try.

- **Prevention:** In `writing-plans`, when a step is `[HUMAN]`-run and has >1 command, emit it as a script-file + a one-line invocation, never as a pasteable multi-line block. Grep the plan for `[HUMAN]` steps containing `&&` or a trailing `\` ‚Äî rewrite each to a script invocation.

### 2026-06-19 ‚Äî `git mv` + symlink at the IDENTICAL old path registers as add+typechange, not a rename (history check needs a blob compare)

- **Cause-tag:** `git-symlink-shadows-rename`
- **What happened:** Verifying "history preserved" after the hook move, `git diff --cached --find-renames` returned ZERO renames and `git status` showed ` T .claude/hooks/<name>.sh` (typechange). Alarming ‚Äî the skills migration had shown clean `R100` renames. Root cause: for skills the moved unit was a *directory* and the symlink took a *new* leaf path, so the old file path was deleted ‚Üí git paired it as a rename. For hooks the symlink occupies the **exact same path** as the old file (`.claude/hooks/<name>.sh`), so the old path is not deleted ‚Äî it becomes a symlink (mode 120000) ‚Äî and git sees "new file added under `hooks/` + old path typechanged to symlink", which rename detection does not pair. History is still intact: `git rev-parse HEAD:.claude/hooks/<name>.sh` == `git rev-parse :hooks/<cat>/<name>.sh` (identical blob `3cee0a7‚Ä¶`), so `git log --follow` traces it by content.
- **Fix / rule:** "No `R` in `git status`" does NOT mean history was lost. When a migration places a symlink at the same path a real file used to occupy, verify history by **blob-identity** (`git rev-parse HEAD:<old>` == `git rev-parse :<new>`), not by rename detection. The end state (real file added at new path, old path staged as a 120000 symlink) is correct and committable.
- **Prevention:** For any "move + symlink-back" migration, pick the history check by shape: dir-move with symlink at a new name ‚Üí expect `R100`; file-move with symlink at the identical old path ‚Üí expect add+typechange, confirm via blob-equality instead.

### 2026-06-19 ‚Äî Patching docs for a layout change, baked the volatile `<category>` level into 7 places where only the stable invariant matters

- **Cause-tag:** `doc-over-coupled-volatile-detail`
- **What happened:** Updating the docs for the categorized-skill-layout migration, I wrote `skills/<category>/<name>/` (and once enumerated all five category names, and once injected a full path into a *role-level* layer reference ‚Äî `framework.md` "the layers of a skill change") across 7 locations: glossary row #1, `framework.md` L10, `skill-routing-sync.md` L22+L62, root `CLAUDE.md` L19, `.claude/CLAUDE.md` L17+L32. But `<category>` is the one dimension the design explicitly made **volatile** ‚Äî cheap to re-categorize, deliberately NOT leaked into routing. The stable invariant is `skills/` + the flat symlink `.claude/skills/<name>`, with `name === leaf dir === symlink === routing key` (category does not participate). Coupling docs to the volatile level creates drift debt: re-categorize a skill and these cells/sentences go stale. The owner caught it twice (framework, then glossary, then skill-routing-sync) before I swept the whole class.
- **Fix / rule:** When patching docs for a structural change, reference the **stable invariant, not the volatile dimension**, and match the prose's existing abstraction level ‚Äî a layer-*role* reference ("SKILL.md frontmatter") carries no path; a location cell uses `skills/` (or a `skills/*/<name>/` glob), never the specific taxonomy level; never enumerate a taxonomy designed to change. Swept all 7 to `skills/` / `skills/*/<name>/`.
- **Prevention:** After a layout-driven doc edit, grep the volatile token across **all** edited docs in one pass (`grep -rn '<category>' <edited docs>`); any hit in a cell/sentence not specifically about that dimension is over-coupling ‚Äî strip to the stable form then, not after the owner flags each file. Fix the whole class at once.

### 2026-06-19 ‚Äî Claude Code follows symlinked skill directories for discovery (undocumented ‚Äî verified, not assumed)

- **Cause-tag:** `undocumented-platform-behavior`
- **What happened:** Reorganizing the vault's 15 skills into category folders (`skills/<category>/<name>/`) with flat symlinks `.claude/skills/<name>` ‚Üí `../../skills/<category>/<name>`, the load-bearing question was whether Claude Code discovers a skill through a **symlinked** directory. The official docs (code.claude.com) cover symlinks only for *rules*, and say nothing about symlinks for *skills* ‚Äî readiness-review flagged this as an unverified go/no-go. Resolved empirically: with the real skill at `skills/apply-chain/grilling/` and a `mode 120000` symlink at `.claude/skills/grilling`, a subagent in a fresh context invoked `Skill(grilling)` and the body loaded (reporting its base dir as the symlink path). Discovery follows the symlink transparently.
- **Fix / rule:** When a design rests on platform behavior the vendor docs do NOT document, do not proceed on "it probably works" ‚Äî run a minimal empirical probe (here: one symlinked skill + a fresh-context `Skill` invocation) before building the full migration on it. Also neutralize the risk *structurally* where possible: variant B keeps `.claude/skills/` containing **only** symlinks (no category dirs), so even if discovery were recursive there is no second copy to double-register ‚Äî the uncertainty is engineered out, not just tested around.
- **Prevention:** For any change whose correctness depends on undocumented tool/platform behavior, add a one-shot probe step to the plan that reproduces the exact mechanism in isolation, and prefer a structure that is correct under both possible behaviors. Treat "docs are silent" as "must verify", never as "safe by default".

### 2026-06-19 ‚Äî Reference-link validator regex flagged an illustrative inline example as a broken link (false positive)

- **Cause-tag:** `self-check-format-drift` (2nd instance)
- **What happened:** During final acceptance of the skill-layout migration, an ad-hoc reference-link validator (`grep -oE '\]\(([^)]+\.md)\)'`) reported `BROKEN-LINK: skills/authoring/writing-rules/SKILL.md -> ./error-handling.md`. Investigation: that string is an **illustrative inline example** of the cross-linking pattern in the skill's prose ‚Äî `Cross-link siblings with relative links (`[error-handling](./error-handling.md)`)` ‚Äî not a navigational link the skill relies on. The file never existed; the "link" was present in HEAD, so the migration neither introduced nor broke anything. The validator gave a wrong verdict by not distinguishing a demonstrated-pattern example from a real link.
- **Fix / rule:** Same family as the promotion-debt scan drift below: a verification-step regex over markdown must account for real markdown structure. A reference-link checker must exclude links that are illustrative ‚Äî inside backtick inline spans, inside fenced example blocks, or explicitly marked illustrative ‚Äî before declaring a broken link, or it produces false positives that erode trust in the check. Did NOT "fix" the writing-rules example (pre-existing, out of scope for this migration ‚Äî like the `improve-codebase-architecture` routing gap); recorded it instead.
- **Prevention:** When writing/automating the vault's "reference links resolve" validator, skip any `](‚Ä¶)` occurrence whose surrounding context is a backtick inline span or a fenced block; only treat top-level prose links as navigational. Re-confirm a flagged "broken link" by reading its line before acting on it.

### 2026-06-19 ‚Äî `lessons-learned-protocol`'s own promotion-debt scan silently matched nothing (format drift)

- **Cause-tag:** `self-check-format-drift`
- **What happened:** The scan command documented in `lessons-learned-protocol/SKILL.md` was `grep -oE 'Cause-tag\*\*:‚Ä¶'` ‚Äî it matches `Cause-tag**:` (template format), but the real entries are written `**Cause-tag:**` with the tag in backticks, and the file's intro prose says `` `Cause-tag` ``. Run literally on the live log it returned **empty** ‚Üí a false "no promotion debt", so a cluster could cross the threshold of 3 unnoticed. Three cosmetic formats of one field had drifted apart and the self-check matched none of the actual entries.
- **Fix / rule:** A field-scraping regex in a skill's own verification step must be anchored to the field's stable structural marker and tolerant of cosmetic markdown variance ‚Äî not pinned to one punctuation layout. Fixed to anchor on the **list-item line start** (`^[[:space:]]*-[[:space:]]+\*\*Cause-tag[^[:alnum:]]+[a-z0-9-]+`): absorbs colon-inside/outside and optional backticks, and the line anchor excludes prose that merely quotes the field (a body-only anchor still matched a lesson that discussed `Cause-tag` in its Fix text ‚Äî a real false positive caught only because the log itself now contains such prose).
- **Prevention:** Author/verify any "run this to self-check" command against the REAL artifact at write time, never assume it matches the template; for markdown field scrapes, anchor on the field's **line-start list marker** (not just the bold span, which prose can quote) and allow `[^[:alnum:]]+` between marker and value.

### 2026-06-19 ‚Äî A RED scenario that foregrounds tool choice contaminates the test and yields a false GREEN

- **Cause-tag:** `red-scenario-contamination`
- **What happened:** Diagnosing the `lessons-learned-protocol` bypass, the first RED prompt asked the subagent to "state the exact first tool call you would make". It complied via `Skill` ‚Äî a false GREEN. The premise (the agent writes the lesson via direct `Edit`) only reproduced when the scenario was naturalistic: a real task with the capture incidental and **no tool named**, where the agent ran `Read‚ÜíRead‚ÜíEdit`, bypassing the skill. Same agent, same manual ‚Äî only the prompt's framing differed.
- **Fix / rule:** A RED/baseline subagent prompt must never name, or ask the agent to reason about, the tool/skill under test ‚Äî that cue is the thing you are trying to observe emerge (or not) on its own. Give a realistic task, let the behaviour happen, and read it from a post-hoc tool list or `git diff`, not from a "which tool would you use" question. This is `writing-great-skills`' inversion test applied to the *scenario*: if foregrounding the choice flips the result, the scenario, not the skill, decided it.
- **Prevention:** Before running a baseline, grep your own scenario prompt for the tool/skill name under test (`Skill`, `Edit`, the skill's name) ‚Äî any hit in the instruction is contamination; strip it and re-run.

### 2026-06-19 ‚Äî "Add a verify/review phase" defaults to a no-op; the binding value is the efficacy test, not another correctness pass

- **Cause-tag:** `skill-value-vs-noop` (2nd instance)
- **What happened:** Asked to add a "Review phase" to `writing-rules` that tests a freshly-written rule "meets all conditions". The literal reading is a *correctness* review. ### 2026-06-19 ‚Äî A skill's "principles" are a no-op for a strong model; the binding value is lexicon + deliverable shape + skipped process steps

- **Cause-tag:** `skill-value-vs-noop`
- **What happened:** RED baseline for migrating `codebase-design` / `improve-codebase-architecture` ‚Äî 4 cold subagents on well-framed design prompts reached the skills' *principles* unaided (deletion-test reasoning, deep/shallow detection, inject-don't-create, fakes>mocks, "no seam until two adapters", domain-shaped narrow ports). A skill teaching those to a strong model proves little (inversion test: it complies WITHOUT the skill). What it reliably FAILED, 4/4: vocabulary drift (every agent wrote the explicitly-forbidden "boundary", plus "layers/wrappers/shell/core" ‚Äî no fixed lexicon), no structured visual deliverable when asked to "present findings", skipped the explore‚Üípresent‚Üígrill process, and produced ONE interface design instead of design-it-twice.
- **Fix / rule:** Calibrate GREEN to the *reproduced* failure, not the source's stated purpose. For these skills the defensible value = enforced shared vocabulary + the structured deliverable + the process steps a model skips ‚Äî keep those load-bearing, treat principle-prose as light scaffolding. Vocabulary drift is a *shaping* failure ‚Üí fixed-lexicon recipe ("use exactly X; never substitute Y"), not prohibitions (per `writing-great-skills` "Match the Form to the Failure").
- **Watch:** if this recurs (a third skill whose principle-content is no-op for strong models) ‚Üí promote to a rule on scoping skill value to reproduced failures.

### 2026-06-19 ‚Äî Naive fence-toggle corrupts markdown-in-markdown when auto-fixing

- **Cause-tag:** `markdown-fence-counting`
- **What happened:** A bulk fixer that added a language to "bare opening fences" treated every ` ``` ` as a toggle. In template files that wrap example fenced blocks in a four-backtick fence (` ````markdown ` ‚Ä¶ ` ```` `), the inner three-backtick fences are literal content; naive toggling desynced and appended `text` to *closing* fences.
- **Fix / rule:** Parse fences per CommonMark ‚Äî a fence opened with N backticks closes only on a line of ‚â•N backticks with no info string; inner shorter fences are content. Same rule for skipping tables inside example blocks. Codified in [rules/common/markdown-style.md](rules/common/markdown-style.md) (Fenced-code bullet + Edge Cases).
- **Also:** Python `glob('**/*.md')` skips dot-directories (`.claude/`) ‚Äî use `os.walk` for repo-wide markdown sweeps.
