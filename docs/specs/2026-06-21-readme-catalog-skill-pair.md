# Spec — README catalog skill pair (`bootstrapping-readme` / `auditing-readme`)

> Source: approved S6 design from `vault-expansion-audit.md`, Decisions D1–D9 (fixed input, not re-litigated).
> Status: spec draft for review. Date: 2026-06-21.

## Goal

Add a foundation skill **pair** — `bootstrapping-readme` (generates) and `auditing-readme` (checks drift) — that maintains a human-facing README skills catalog inside a **managed block**, agnostic over any skills-bearing repo, sourcing every row from skill frontmatter on disk.

## Scope

- Two new skills under `skills/foundation/`, mirroring the `bootstrapping-glossary` / `auditing-glossary` pair (SKILL.md + `assets/` + duplicated `references/catalog-derivation.md`).
- A single shared **catalog-derivation contract** (discovery → category → kind → description → row), encoded identically in both skills, so generator and auditor agree by construction.
- The managed README block format (markers, headings, table schema).
- `auditing-readme` drift criteria + per-finding report (mirrors `auditing-glossary`).
- Two `.claude/skills/` flat symlinks and two `skills-routing.json` entries with triggers.
- A committed root `README.md` for this vault, produced by running `bootstrapping-readme` (dogfood output, per D8).

## Out of scope

- Any hook (PostToolUse/Stop) that auto-regenerates or nudges (D3 — no hook dependency; a separate candidate).
- Whole-file README ownership — the skill owns only the marker block; intro/install/badges/license are human-authored (D2).
- A consumer-app project-README generator (D1 — rejected framing).
- A `summary:` frontmatter field across skills (rejected in grilling Q5 as scope creep).
- Fixing the stale glossary claim that `writing-skills` is `disable-model-invocation: true` (it is `false`) — real drift, but owned by `auditing-glossary`; recorded as a follow-up.
- Promoting/removing the S6 row in `vault-expansion-audit.md`.

## Contracts

### Managed block format (the artifact `bootstrapping-readme` writes)

````text
<!-- skills:start -->
<!-- Generated by bootstrapping-readme — do not edit by hand; rerun the skill. -->

### apply-chain

| Skill | What it does | Kind |
| --- | --- | --- |
| [grilling](skills/apply-chain/grilling/SKILL.md) | Turn a fuzzy idea into a shared, concrete design. | auto-routed |

### entrypoints

| Skill | What it does | Kind |
| --- | --- | --- |
| [sdd](skills/entrypoints/sdd/SKILL.md) | Run the full SDD pipeline end-to-end. | alias |
<!-- skills:end -->
````

- The `## Skills` H2 that owns the block is **human-authored, outside** the markers. Block content = `###` category subsections + GFM tables only.
- Tables obey `markdown-style` (spaced delimiter row `| --- | --- |`, blank line around headings/tables).
- Flat repos (no category dirs): a single table, no `###`.

### Catalog-derivation contract (encoded in `references/catalog-derivation.md` in BOTH skills)

```text
ROOT        = `skills/` if it exists and any SKILL.md sits in a sub-directory of it;
              else the directory that directly holds the SKILL.md files.
              If `skills/` exists, ROOT is `skills/` regardless of stray SKILL.md elsewhere.
DISCOVERY   = all `**/SKILL.md` under ROOT. Skill id = frontmatter `name`.
CATEGORY    = first path segment of (skill-dir relative to ROOT).
              `foundation/auditing-readme` → `foundation`; depth >1 keeps the FIRST segment.
              Skill dir directly under ROOT → uncategorized (flat).
              No skill has a category segment → flat render (single table, no `###`).
ROW LINK    = repo-root-relative path to the source SKILL.md (`skills/<cat>/<name>/SKILL.md`).
KIND        = `alias`        if CATEGORY == `entrypoints`
              `auto-routed`  else if `name` is a key in `.claude/skills-routing.json` .skills
              `user-invoked` otherwise
              (D4's "reference skills flagged" requirement is satisfied here:
               a reference/methodology skill — not routed, not an alias — renders
               as `user-invoked`, which is D4's literal flag. There is no separate
               `reference` value; `user-invoked` IS the not-hidden flag for both
               reference skills and any other human-invoked skill.)
DESCRIPTION = frontmatter `description` (folded to one string), then:
              1. strip from first match of /\s*Triggers?( on)?:.*/is to end (drops trigger tail, incl. RU);
              2. collapse whitespace/newlines to single spaces, trim;
              3. if len > 120, truncate at last word boundary ≤120 and append `…`.
ORDER       = categories alphabetical; rows alphabetical by `name` within a category.
```

### Skill frontmatter (both NEW)

```yaml
# skills/foundation/bootstrapping-readme/SKILL.md
name: bootstrapping-readme
description: >-
  Use when a skills-bearing repo has no README skills catalog yet (or only a
  stub) and you need to generate the human-facing index of its skills from
  frontmatter on disk. Triggers on: "set up the README", "generate the skills
  catalog", "bootstrap the README", "build the skills index".
allowed-tools: Read, Grep, Glob, Write, Edit
```

```yaml
# skills/foundation/auditing-readme/SKILL.md
name: auditing-readme
description: >-
  Use to check whether the README skills catalog still matches the skills on
  disk (names, descriptions, categories, links, kind) and to correct it when it
  drifts. Triggers on: "audit the README", "is the skills catalog accurate",
  "README drift", "check the skills index", "update the README catalog".
allowed-tools: Read, Grep, Glob, Edit
```

### `auditing-readme` drift criteria → per-finding report

```text
(1) markers present and well-formed (one pair, start before end)         finding #1 (blocks all else)
(2) every on-disk SKILL.md appears exactly once in the block
(3) category grouping + ordering match disk
(4) each row description matches the current derived description
(5) each row link resolves to an existing SKILL.md
(6) each row kind matches routing-derived kind
Report = table {finding → what disk shows → status}; recommended disposition
per finding = "regenerate block" (rerun bootstrapping-readme); mirrors
auditing-glossary report shape (assets/audit-report-example.md).
```

## Edge cases (D9 — decided, not deferred)

- **Duplicate `name`** across two SKILL.md → **error** naming both conflicting paths; do not silently merge (name===dir===symlink invariant already forbids it).
- **Missing `name` or `description`** → emit a `malformed frontmatter: <path>` finding row; never crash the walk.
- **Malformed/duplicate markers** (one `start`, two `end`, reversed, nested) → `bootstrapping-readme` **refuses** to edit and reports; `auditing-readme` flags as finding #1 (blocks 2–6).
- **Zero skills under ROOT** → bootstrap writes an empty block with a placeholder comment (`<!-- no skills found -->`); audit passes vacuously.
- **No README / README without markers** → bootstrap creates the file (`# <repo>` H1 + `## Skills` + block) or inserts the block under a `## Skills` heading, leaving existing prose untouched.
- **Description with no trigger tail** (7 vault skills: the 4 `entrypoints/` aliases + `writing-skills` + `codebase-design` + `improve-codebase-architecture`; note `handoff`'s `Triggers:` and `resolving-requirements`' `Russian triggers:` ARE stripped by the regex) → step-1 strip is a no-op; steps 2–3 still apply. Empty/whitespace description after strip → treated as missing (`malformed` finding).

## Verification

No build/test pipeline (skills vault). Proof is:

- **Validators** on both new `SKILL.md` (+ their `references`/`assets`): frontmatter ≤1024, `name` regex `^[a-z0-9-]+$`, every reference/asset link resolves, fences balanced, word count sane. Paste output.
- **Routing/symlink sync** (`skill-routing-sync` checklist): `find skills -name SKILL.md` ↔ `jq '.skills|keys'`; both new symlinks resolve (`readlink`); `jq . .claude/skills-routing.json` valid.
- **RED → GREEN, on GENERATED output** (per `dogfood-generator-sync`, D8):
  - RED: cold subagent (no skill) asked to "create/maintain a README cataloging this repo's skills" — record the failure shape (inconsistent format, skills missing/extra, hand-curated descriptions diverging from frontmatter, no markers).
  - GREEN-bootstrap: with `bootstrapping-readme`, generate the block on this vault; confirm all on-disk skills present, format stable, descriptions match the derivation.
  - GREEN-audit: introduce a drift (rename a skill dir; make one row description stale) and confirm `auditing-readme` flags exactly criteria 2 and 4 with the regenerate disposition.

## Risks

- **Description truncation** may cut a clause awkwardly — mitigated by word-boundary + `…`; acceptable for a one-line catalog. Authors wanting precise copy can tighten the source `description`.
- **Derivation duplicated in two `references/` files** → drift risk between generator and auditor. Mitigated by `dogfood-generator-sync` (move both in lockstep) and by the audit re-deriving from the same documented algorithm. Known, accepted (mirrors `placeholder-keys.md`).
- **ROOT auto-detection heuristic** ("`skills/` with sub-dir skills") could misclassify an unusual layout → mitigated by the explicit flat-render fallback; the consumer repo can state ROOT.
- **`disable-model-invocation` is NOT used for kind** (verified unreliable: `writing-skills` has it `false` yet is user-invoked). A future skill that is meant to be routable but is absent from `skills-routing.json` would render `user-invoked` instead of `auto-routed` — acceptable, and itself a routing-sync defect the other tooling catches. This does not affect D4 conformance: reference and other human-invoked skills both correctly carry the `user-invoked` flag.
